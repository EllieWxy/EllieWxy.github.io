node使用pm2

发表于 2020-05-19

pm2是一个带有负载均衡功能的node进程管理器，可以用它来跑node进程。可以帮助应用在崩溃后实现自动重启。

pm2 list

用来查看当前跑了哪些进程

pm2 logs

用来打印日志查看

保证常驻应用进程稳定运行：
  • 定时重启，应用进程运行时间久了或许总会产生一些意料之外的问题，定时可以规避一些不可测的情况；
  • 最大内存限制，根据观察设定合理内存限制，保证应用异常运行；
  • 合理min_uptime，min_uptime是应用正常启动的最小持续运行时长，超出此时间则被判定为异常启动；
  • 设定异常重启延时restart_delay，对于异常情况导致应用停止，设定异常重启延迟可防止应用在不可测情况下不断重启的导致重启次数过多等问题；
  • 设置异常重启次数，如果应用不断异常重启，并超过一定的限制次数，说明此时的环境长时间处于不可控状态，服务器异常。此时便可停止尝试，发出错误警告通知等。

跨域

发表于 2020-05-19

为什么要进行跨域

由于浏览器同源限制的策略，非同源的网站在请求时需要进行跨域。

##跨域的常见方法

CORS（跨域资源共享）

浏览器自动完成，不需要用户的参与。关键在于服务器的设置。

简单请求/非简单请求

简单请求可以直接发出CORS请求，非简单请求需要发送预检请求，在服务器同意后再发送一次请求。

CORS请求

1.浏览器会在请求头上添加Origin字段，来表明自己来自哪个源（协议+主机+端口）

2.如果Origin中的字段在Access-Control-Allow-Origin内，那么就在response的header里多出几个头信息字段.

  • Access-Control-Allow-Origin 该字段是必须的
  • Access-Control-Allow-Credentials 可选，表示是否允许携带cookie
    • 如果要携带cookie，还需要在ajax中设置xhr.withCredentials = true;，否则浏览器不会带上cookie
    • 同时，携带cookie要求Access-Control-Allow-Origin字段不能为*，必须指定明确的，与请求网站一致的域名
    • 同时，cookie仍然遵守同源策略，只会携带Access-Control-Allow-Origin中包含的域名的cookie
  • Access-Control-Expose-Headers 可选，表示服务器还允许getResponseHeader方法拿到哪些header

3.如果没有这些字段的话说明本次请求不符合跨域请求，使用xhr的onerror方法来捕获

服务器的回应每次都会带上Access-Control-Allow-Origin

预检请求

  1. 浏览器发现这是一个非简单请求，就发送一个预检请求。

    (1).请求方法Option，关键字是Origin

    (2).必须携带的字段

    • Access-Control-Request-Method 列出请求的方法

    • Access-Control-Request-Headers 列出请求所添加的额外的头信息

浏览器的回应

  1. 请求未通过，则会返回一个正常的http请求，但不会有关于CORS的头部信息，则浏览器就知道未通过。可以在xhr的onerror中捕获
  2. 请求通过，在返回信息中包含以下字段
    • Access-Control-Allow-Origin 必需
    • Access-Control-Allow-Methods 表示服务器允许的所有请求方法，不单局限于预检请求中发送的，这是为了避免发送多次预检请求
    • Access-Control-Allow-Headers 请求时包含Header则此字段必需，表示服务器允许的所有头部字段
Access-Control-Allow-Origin

在后端设置此字段，在response的header信息中添加。

JSONP

通过动态插入script脚本的方式来进行跨域，基本上所有浏览器都支持，但本方法只能使用GET请求。

反向代理

  1. nginx为例，前端页面在127.0.0.1:8001，后端服务在127.0.0.1:3001
  2. 此时启动一个代理服务器在3002端口
  3. 然后让前端所有的请求都发往3002，如果nginx匹配到请求字段中有/api，那么就说明这是发往后端的请求，将其发往127.0.0.1:3001来代理前端进行请求。再将请求得到的资源发回前端。

PostMessage

是一个进行窗口之间跨域请求的api，在h5中新增。

事件循环

发表于 2020-05-15

什么是事件循环

事件循环，实际上是浏览器执行任务的机制。众所周知，js是单线程的，但是有时候如果要进行一些需要等待的操作，也就是异步操作，不可能一直阻塞等着结果的返回。所以就出现了事件循环。

事件循环的机制

首先，script中执行的代码分为同步和异步。同步代码直接执行，异步代码要在满足条件后被扔到相应的task queue中。

在当前的执行栈为空后，浏览器会先去检查微任务队列，并将其中的任务执行。微任务队列为空后，去检查宏任务队列，如果有任务就执行。

宏任务&微任务

先执行微任务队列中的任务，再执行宏任务队列中的任务

微任务

拓展-node中的事件循环

![image-20200520111252990](/Users/EllieWxy/Library/Application Support/typora-user-images/image-20200520111252990.png)

node中的事件循环分为了六个阶段，顺序是从外部输入数据开始，之后poll-> check->close callbacks -> timers -> I/O callbacks -> idle,prepare-> poll

每个阶段都有它负责的事件。

  • timers 阶段：这个阶段执行timer（setTimeout、setInterval）的回调

  • I/O callbacks 阶段：处理一些上一轮循环中的少数未执行的 I/O 回调

  • idle, prepare 阶段：仅node内部使用

  • poll 阶段：获取新的I/O事件, 适当的条件下node将阻塞在这里

  • check 阶段：执行 setImmediate() 的回调

  • close callbacks 阶段：执行 socket 的 close 事件回调

timers

执行定时器（setTimeout,setInterval)

poll

主要干两件事

  1. 如果定时器已经超时，就返回定时器执行相应的回调函数。
  2. 执行I/O回调
    • （所以大多数的I/O会在poll阶段执行，I/O callback负责执行上轮未执行的少部分回调）
  3. 如果没有要处理的定时器
    • poll不为空的话，就执行poll中的回调函数
    • poll为空
      • 存在setImmediate，从poll阶段进到check阶段
      • 不存在setImmediate，在poll阶段等待直到有事件为止
check

执行setImmediate

nextTick

在上一个阶段结束后，下一个阶段执行前进行执行

node 11 vs node 10

在node 11的机制中，执行完一个宏任务，就会去检查微任务队列并清空。

而在node10以前的机制中，需要执行完一队宏任务，才会去检查微任务队列并清空。

浏览器渲染

发表于 2020-05-15

css会阻塞DOM树吗

css不会阻塞DOM树的解析，但是会阻塞DOM树的渲染

css会阻塞js吗

会，因为渲染线程和js线程会相互阻塞

js会阻塞渲染吗

会。

可以用defer或async解决

defer
  • 在html解析完成后再执行
async
  • 加载完后立即执行（仍然会阻塞html渲染

浏览器渲染的两个函数

1.DOMContentLoaded

表示DOM内容加载完毕

2.onload

页面上所有的资源（图片，音频，视频等）被加载以后才会触发load事件

onload事件会在DOMContentLoader事件之后触发

响应式布局是什么

响应式布局就是实现不同屏幕分辨率的终端上浏览网页的不同展示方式，即在电脑、手机、平板上会有不同的展示页面。

传统的布局方式是根据不同的尺寸写不同的页面，响应式布局要求一个网站一套就可以

响应式布局的实现方案

1.设置meta标签

表示使用设备的宽度作为视图宽度

2.设置媒体查询

在屏幕宽度小于980px时会使用这一套css

3.使用vh/vw

vh/vw是相对于视窗的尺寸

4.使用百分比

在一些情况下容易炸（要慎重使用）

5.使用一些新特性或框架

例如

  • Flex弹性布局，兼容性较差
  • Grid网格布局，兼容性较差
  • Columns栅格系统，往往需要依赖某个UI库，如Bootstrap
